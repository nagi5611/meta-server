# プレイヤー同期システム再構築計画

## 現状の問題（事実）

プレイヤー同期システムが壊れており、再構築が必要。

## 現状コードの同期構造（事実）

### `nodeserver_/server.js`の現状
- `playerMove` イベントを受信したら即座に `broadcast` する構造
- リアルタイム転送（受信即配信）

### `metaverse-simple/server.js`の現状
- 33msごとに `io.emit('players-update', playersArray)` で全員に配信
- サーバーtick方式を採用

## 再構築後の同期仕様（確定要件）

### 基本方針
- **クライアント→サーバー**: max30Hz送信
- **サーバー→クライアント**: 33ms（max30Hz）のtickでまとめて配信
- **データ形式**: position + Euler + quaternion + timestamp
- **timestamp処理**: 古いtimestampは破棄
- **ルーム単位**: 同一ルーム内のプレイヤーのみ同期（ルーム一律max30Hz）
- **距離による調整**: 今回は実装しない（将来フェーズ）

### データフロー

```
クライアント（30Hz以下）
  ↓ playerMove送信
サーバー受信（Ingress）
  ↓ timestamp検証→最新stateのみ保持
ルーム別プレイヤーstate
  ↓ 33ms tick
スナップショット生成（ルーム単位）
  ↓ playersSnapshot配信
ルーム内全クライアントへ配信
```

## 実装する同期コンポーネント

### 1. Ingressレイヤー（クライアント→サーバー）

**責務**: クライアントからの位置更新を受信し、最新stateのみ保持

**処理内容**:
- `playerMove` イベント受信
- 受信データ構造:
  ```json
  {
    "position": { "x": 0, "y": 0, "z": 0 },
    "rotation": { "x": 0, "y": 0, "z": 0 },
    "quaternion": { "x": 0, "y": 0, "z": 0, "w": 1 },
    "timestamp": 1234567890123
  }
  ```
- timestamp検証: サーバー側の最新timestampより古い場合は破棄
- ルームIDに紐づけてプレイヤーstateを更新

### 2. ルーム別プレイヤーstate管理

**データ構造**:
```javascript
roomStates = {
  "roomId": {
    players: {
      "socketId": {
        position: { x, y, z },
        rotation: { x, y, z },
        quaternion: { x, y, z, w },
        timestamp: Number,
        username: String
      }
    }
  }
}
```

### 3. サーバーTickシステム

**処理内容**:
- `setInterval(tickFunction, 33)` でtickループ起動
- 各ルームごとに処理:
  1. ルーム内の全プレイヤーstateを配列化
  2. スナップショット生成
  3. `io.to(roomId).emit('playersSnapshot', snapshot)` でルーム内配信

**配信データ構造**:
```json
{
  "timestamp": 1234567890123,
  "players": [
    {
      "id": "socketId",
      "username": "プレイヤー名",
      "position": { "x": 0, "y": 0, "z": 0 },
      "rotation": { "x": 0, "y": 0, "z": 0 },
      "quaternion": { "x": 0, "y": 0, "z": 0, "w": 1 }
    }
  ]
}
```

### 4. クライアント側送信レート制限

**処理内容**:
- クライアント側で送信間隔を33ms以上に制限（max30Hz）
- 実装例:
  ```javascript
  let lastSendTime = 0;
  const SEND_INTERVAL = 33; // ms
  
  function sendPlayerUpdate(position, rotation, quaternion) {
    const now = performance.now();
    if (now - lastSendTime < SEND_INTERVAL) return;
    
    socket.emit('playerMove', {
      position,
      rotation,
      quaternion,
      timestamp: Date.now()
    });
    
    lastSendTime = now;
  }
  ```

## 実装手順

### ステップ1: サーバー側Ingress実装

**対象ファイル**: `nodeserver_/server.js` または `metaverse-simple/server.js`

**作業内容**:
1. ルーム別プレイヤーstateストレージを初期化
2. `playerMove` イベントハンドラーを以下に変更:
   - socketのルームIDを取得
   - 受信timestampと既存timestampを比較
   - 新しい場合のみstate更新
   - 即座にbroadcastしない（tick配信に任せる）

### ステップ2: サーバーTickシステム実装

**作業内容**:
1. `setInterval` でtickループ起動（33ms）
2. 全ルームをループ処理
3. 各ルームのプレイヤーstateをスナップショット化
4. `io.to(roomId).emit('playersSnapshot', snapshot)` で配信

### ステップ3: クライアント側受信処理実装

**対象ファイル**: クライアントのネットワーク管理スクリプト

**作業内容**:
1. 既存の個別プレイヤー更新イベント（`userConnected`, `playerMove`等）を削除または無効化
2. `playersSnapshot` イベントリスナーを実装
3. 受信したスナップショットから全プレイヤー情報を更新
4. 自分のsocketIdは除外して他プレイヤーのみ表示更新

### ステップ4: クライアント側送信レート制限実装

**作業内容**:
1. 送信間隔チェック機構を追加（33ms minimum）
2. position/rotation/quaternion + timestampを含めて送信
3. 既存の高頻度送信コードを置き換え

## 既存コードとの互換性

### 保持するイベント
- `userConnected`: 入室通知（初回ルーム入室時のみ使用可能）
- `userDisconnected`: 退室通知
- `chatMessage`: チャット（同期システムとは独立）
- `sendEmoji`: エモート（同期システムとは独立）

### 削除/置き換えるイベント
- `playerMove` の即時broadcast → Ingress + tick配信に変更
- 個別プレイヤー更新イベント → `playersSnapshot` 一本化

## 監視指標（フェーズ3で実装）

同期システムの正常性を監視するための指標:
- ルーム別 `playerMove` 受信数/秒
- ルーム別 `playersSnapshot` 配信数/秒（≒30回/秒）
- timestamp破棄率（古いデータの割合）
- クライアント送信timestamp - サーバー受信時刻の差（遅延推定）
- 送受信バイト/秒

## 注意事項

### 今回実装しないもの
- 距離によるHz調整（同一ルーム一律30Hz）
- 複数Nodeプロセスでの分散（単一プロセス）
- 補間/予測（クライアント側で実装する場合は別途）

### 将来拡張の余地
- 距離に応じた送信Hz削減（遠いプレイヤーは10Hzなど）
- サーバー側での簡易な位置検証（チート対策）
- 状態圧縮（JSON→バイナリなど）
